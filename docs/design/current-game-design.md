# ADOM-Clone Current Design (Phase 3 Baseline)

## 1. Goal of the Current Baseline

This baseline provides a playable vertical slice with:

- tile-based overworld and dungeon maps,
- turn-based movement and combat,
- inventory pickup/use/drop loop,
- simple monster AI,
- save/load scaffolding,
- pygame rendering and controls.

The code is organized to keep game rules in `core` and UI/input in `client`.

## 2. High-Level Architecture

### 2.1 Layering

- **Domain/Core** (`src/adom_clone/core`)
  - ECS data model and component storage
  - world map models and generators
  - game session orchestration (turns, combat, inventory, persistence)
- **Client** (`src/adom_clone/client`)
  - pygame event loop
  - key mapping to domain actions
  - rendering and HUD
- **Entry point** (`src/adom_clone/main.py`)
  - starts the pygame client

### 2.2 ECS Pattern

- **Entity**: integer ID generated by `ECSStore`.
- **Components**: dataclasses such as `Position`, `Fighter`, `Inventory`, `Monster`.
- **Systems**: implemented procedurally inside `GameSession` methods.

Current ECS store is intentionally minimal: component-type-indexed dictionaries with typed getters.

## 3. Core Runtime Model

## 3.1 `GameSession` Responsibilities

`GameSession` is currently the orchestration hub and owns:

- map instances (`overworld`, `dungeon`, `current_map`),
- player entity and initial content spawn,
- action queue and turn progression,
- combat/inventory processing,
- transitions between overworld and dungeon,
- run metrics (`turn_count`, `kill_count`),
- bounded message log,
- save/load serialization.

### 3.2 Turn Flow

1. client queues one `GameAction` from input,
2. `advance_turn()` dequeues and applies player action,
3. if action was effective, increment turn counter,
4. execute monster phase,
5. update message log and game-over state as needed.

### 3.3 Movement and Collision

- map passability is checked first,
- blocking entity lookup determines collision,
- if target is a monster, movement converts into melee attack,
- otherwise player position updates,
- map transition logic runs after successful move.

### 3.4 Combat

- deterministic damage model: `max(1, attacker.power - defender.defense)`,
- melee only,
- on HP <= 0: either game over (player) or component cleanup (monster death),
- kill count increments on monster death.

### 3.5 Inventory

- pickup requires item at player tile and free capacity,
- picked items are detached from world by removing `Position` and `OnMap`,
- use action currently supports healing consumables,
- drop puts item back at player tile on current map.

### 3.6 Persistence

- `to_save_data()` serializes all entities carrying persisted components,
- `save_to_file()` writes compact JSON,
- `from_save_data()` validates shape/types and rebuilds ECS state,
- `load_from_file()` reads JSON and delegates to validator/rehydrator.

The save contract is versioned with `"version": 1` and includes run-level fields.

## 4. World/Content Design

- **Overworld**: bordered area with central dungeon entrance.
- **Dungeon**: enclosed room layout with fixed exit stairs.
- **Initial content**:
  - player in overworld,
  - monsters in dungeon,
  - healing items in both maps.

Generation is deterministic and hand-authored enough to support predictable tests.

## 5. Client/UI Design

- tile renderer with per-tile colors,
- simple glyph-less entity rendering via colored rectangles,
- HUD with map, HP, turns, kills, controls, inventory, and latest messages,
- hotkeys:
  - movement: WASD/arrows
  - wait: `.` / space
  - pickup: `G`
  - use slot: `1..9`
  - drop last: `R`
  - new run: `N`
  - save/load: `F5` / `F9`
  - quit: `ESC`

## 6. Testing Strategy (Current)

Current tests cover:

- ECS component lifecycle,
- map transitions,
- combat + inventory interactions,
- save/load round trip and missing-save behavior.

Validation workflow: `pytest`, `ruff`, `mypy`.

## 7. Known Architectural Constraints

- `GameSession` currently mixes multiple systems and will eventually need decomposition.
- No initiative/speed model; all monsters currently act once after each valid player action.
- No FOV memory, equipment slots, stats/skills progression, quests, factions, or procedural depth scaling yet.
- Save schema migration strategy is not implemented beyond a version field.
